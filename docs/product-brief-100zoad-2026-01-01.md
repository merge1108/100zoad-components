## 요약 (Executive Summary)

부동산 분양 홍보 랜딩페이지를 위한 규격화된 웹 컴포넌트 시스템을 개발합니다. Cloudflare Pages와 Shadow DOM을 활용하여 아임웹 기반 웹디자인 및 개발 시간을 대폭 단축하고, 코드 지식이 없는 팀원도 안전하게 스타일과 문구를 수정할 수 있도록 합니다. 온라인 마케팅 대행사의 2명 팀(오너, 개발자/디자이너)이 사용하며, 클라이언트 1인당 맨아워를 줄이고 저비용 소규모 비즈니스 모델을 유지하는 것이 핵심 목표입니다.

---

## 문제 정의 (Problem Statement)

### 문제

1. **디자인 전문성 부족**: 클라이언트로부터 수주한 랜딩페이지 제작 시 디자인 전문성이 부족하여 품질 향상에 한계
2. **아임웹 기본 스타일의 한계**: 아임웹 기본 스타일이 아름답지 않아 고급스러운 랜딩페이지 제작 어려움
3. **반복 작업으로 인한 시간 낭비**: 매 프로젝트마다 헤더, 푸터, 입력폼 등을 반복적으로 제작하여 시간 소모
4. **코드 지식 부재**: 사내 직원들(개발자 제외)의 코드 지식이 거의 전무하여 수정 불가능
5. **개발자 의존성**: 개발자 부재 시 클라이언트의 수정 요청에 대응 불가능

**구체적 사례:**
최근 오너가 시세보다 60% 저렴한 가격에 멀티페이지 랜딩페이지를 수주했으나, 기존 역량(아임웹만 사용)으로는 완성까지 일주일이 소요됩니다. 이를 원페이지와 동일한 수준(4시간)으로 단축하고, 원페이지는 더욱 짧은 시간에 완성할 수 있어야 합니다.

**현재 상황:**
- **멀티페이지**: 일주일 소요, 2달에 1개 제작
- **원페이지**: 4-5시간 소요 (복잡한 경우 8시간), 월 7개 제작

**목표 생산량:**
- **멀티페이지**: 월 5개
- **원페이지**: 월 15개
- **업무 배분**: 총 20개 랜딩페이지를 15일에 제작, 나머지 15일은 다른 업무

### 지금 해결해야 하는 이유

매 프로젝트마다 반복되는 작업으로 시간이 낭비되고 있으며, 효율적인 워크플로우가 절실히 필요합니다. 빠르게 변화하는 시장에서 경쟁력을 유지하려면 생산성 향상이 필수적입니다.

### 미해결 시 영향

이 문제를 해결하지 못하면 더 많은 인력이 필요하게 되어, 저비용 소규모 비즈니스 모델을 유지하는 데 심각한 어려움이 발생합니다. 결과적으로 경쟁력 상실과 수익성 악화로 이어질 수 있습니다.

## 대상 사용자 (Target Audience)

### 주요 사용자 (Primary Users)

**온라인 마케팅 대행사 팀원 (2명):**

1. **개발자/디자이너 (본인)**
   - 역할: 아임웹 웹디자인 및 개발, 영상 편집, 마케팅 집행
   - 책임: Make.com 활용 리드 전송 솔루션 관리, GTM/GA4 태그 삽입 및 구글애즈 전환추적 관리, 리드 및 상담사 관리, 광고 애셋(영상, 배너) 제작, 구글애즈 마케팅 캠페인 개설 및 관리
   - 특징: 유일한 코드 작업 가능 인력

2. **오너**
   - 역할: 영업 및 경영 담당
   - 책임: 웹/영상/배너 검수, 클라이언트 소통, 마케팅 업무 수주 및 영업 관리
   - 특징: 코드 지식 거의 없음, config 기반 수정 가능해야 함

### 부차적 사용자 (Secondary Users)

**클라이언트 (상담사)**
- 관계: 온라인 마케팅 대행을 의뢰하고 비용을 지불하는 고객
- 니즈: 웹사이트 퀄리티가 좋아 리드가 늘어날수록 이득
- 영향: 웹사이트 품질이 클라이언트의 비즈니스 성과에 직접적 영향

### 사용자 니즈

1. **코드 지식 없이도 쉬운 수정**: 비개발자(오너)도 스타일, 문구, 기능을 안전하게 조정 가능
2. **빠른 랜딩페이지 제작**: 멀티페이지 10시간, 원페이지 4시간 이내 완성
3. **일관된 워크플로우**: 모든 프로젝트에서 동일한 방식으로 작업하여 학습 곡선 최소화
4. **고품질 디자인**: 아임웹 기본 스타일보다 고급스럽고 깔끔한 결과물
5. **안전한 배포**: 개발자 부재 시에도 문제없이 운영 가능

## 솔루션 개요 (Solution Overview)

### 제안 솔루션

**Cloudflare Pages + Shadow DOM 기반 웹 컴포넌트 시스템**

아임웹 기본 스타일에서 격리되어 독립적으로 작동하며, main.html의 config를 통해 코드 지식 없이 모든 설정(색상, 문구, 위치, 기능)을 조정할 수 있는 시스템입니다. 바디 섹션은 정적 이미지로 아임웹에 직접 업로드하고, 기능적으로 필수인 컴포넌트들(헤더, 푸터, 입력폼, 퀵메뉴, 내비게이터)만 웹 컴포넌트로 제작하여 재사용성을 극대화합니다.

### 핵심 기능

#### 1. 헤더 컴포넌트
- **main.html에서 관리**
- 원페이지: 스크롤 기능 (앵커 링크)
- 멀티페이지: 페이징 기능 (페이지 이동)
- config를 통한 메뉴 구성, 색상, 로고 조정

#### 2. 푸터 컴포넌트
- **개별 코드위젯**으로 배치
- config를 통한 링크, 문구, 스타일 조정
- 이용약관, 개인정보처리방침 링크 포함

#### 3. 입력폼 컴포넌트
- **개별 코드위젯**으로 배치
- **필수 필드**: 이름, 전화번호 (향상된 전환 추적용)
- **선택 필드**: config에서 필수/선택 여부 조정
- 이용약관, 개인정보처리방침 필수 체크박스 및 팝업
- GTM/GA4/구글애즈 이벤트 송출
- 이벤트 이름 config에서 수정 가능
- HTTP POST로 Airtable에 리드 전송

#### 4. 퀵메뉴 컴포넌트 (PC용)
- **main.html에서 관리**
- config를 통한 버튼 개수, 색상, 문구, URL, 이벤트 설정
- 각 버튼별 개별 이벤트 추적

#### 5. 모바일 내비게이터 컴포넌트
- **main.html에서 관리**
- config를 통한 버튼 개수, 색상, 문구, URL, 이벤트 설정
- 각 버튼별 개별 이벤트 추적
- 반응형: PC에서는 숨김, 모바일에서만 표시

#### 6. 히어로 이미지 빛 애니메이션
- 정적 이미지 위에 애니메이션 레이어 추가
- **햇빛 효과**: 좌측 상단에서 우측 하단으로 쏟아지는 햇빛
- **밑줄 빛무리 효과**: 히어로 타이틀에 밑줄을 긋는 듯한 빛
- config를 통한 위치, 비율 조정

#### 7. main.html Config 시스템
- 모든 컴포넌트의 중앙 집중식 설정 관리
- 컴포넌트별 config 블럭 (필요에 따라 추가/제거 가능)
- 필수 컴포넌트: 헤더, 퀵메뉴, 모바일 내비게이터 (모든 페이지에 존재)
- 선택 컴포넌트: 푸터, 입력폼 (개별 코드위젯으로 필요시 추가)

### 가치 제안

**우선순위:**
1. **제작 시간 단축**: 멀티페이지 일주일 → 10시간, 원페이지 4-5시간 → 4시간
2. **비개발자 친화성**: 코드 지식 없이도 config 수정으로 모든 조정 가능
3. **일관된 워크플로우**: 모든 프로젝트에 동일한 시스템 적용

**차별점:**
- **vs. 순수 아임웹**: 훨씬 고급스럽고 깔끔한 디자인 가능
- **vs. 풀스택 웹개발**: 확연히 낮은 구현 난이도
- **확장 가능성**: 디자인 외주 활용 시 PNG 파일만 받아 아임웹에 삽입하고, 미리 구현된 웹 컴포넌트를 임포트하여 빠른 구현 및 마케팅 시작 가능

## 비즈니스 목표 (Business Objectives)

### 목표

1. **제작 시간 대폭 단축**

2. **생산성 향상**
   - 멀티페이지: 2달에 1개 → 월 5개 (10배 증가)
   - 원페이지: 월 7개 → 월 15개 (2배 증가)

3. **업무 배분 최적화**
   - 랜딩페이지 제작: 월 15일
   - 다른 업무 (마케팅 관리, 애셋 제작 등): 월 15일

4. **저비용 소규모 비즈니스 모델 유지**
   - 추가 인력 채용 없이 생산성 향상
   - 2명 팀으로 경쟁력 유지

### 성공 지표

**주요 지표:**
- 랜딩페이지 제작 시간 추적 (멀티페이지, 원페이지 각각)
- 월별 랜딩페이지 생산량 (멀티페이지 5개, 원페이지 15개 목표)

**보조 지표 (장기적):**
- 리드 전환율 (데이터가 충분히 쌓인 후 의미 있음)

### 비즈니스 가치

**핵심 가치: 클라이언트 1인당 맨아워 감소**

할애하는 시간이 곧 매출로 이어지도록 워크플로우를 간소화하고 효율화하는 것이 가장 중요합니다. 이 프로젝트는 즉각적인 신규 클라이언트 영업이나 고객 생애가치(LTV) 증가에 직접적으로 작용하지는 않지만, 운영 효율성을 높여 동일한 리소스로 더 많은 클라이언트를 서비스할 수 있게 합니다.

- 푸터 컴포넌트 (개별 코드위젯)

**핵심 기능:**
- main.html 기반 config 시스템

- Cloudflare Pages 호스팅 및 배포
- GTM/GA4/구글애즈 이벤트 통합 및 추적

- CMS (콘텐츠 관리 시스템) 기능
- 백엔드 서버 구축
- A/B 테스팅 기능
- 사용자 인증 시스템

### 향후 고려사항

**없음** - 프로젝트 안착이 최종 목표입니다. 장기적인 확장 계획보다 현재 시스템의 성공적인 정착과 안정화에 집중합니다.

**시간 제약:**
- MVP 출시 마감: 2026-01-01 (오늘)
- 기술적/보안적 안정화: 2026-01-05까지 (4일 이내)
- 랜딩페이지 제작 데드라인: 원페이지 4시간, 멀티페이지 10시간

**예산/인력 제약:**
- 개발 가능 인력: 1명 (본인만 코드 작업 가능)
- 저비용 소규모 비즈니스 모델 (추가 인력 채용 불가)
- 최소 비용으로 최대 효율 달성 필요

- 아임웹 플랫폼 의존성 (코드위젯 제약 사항 준수)

**규제/법적 제약:**

### 가정사항

1. **아임웹 코드위젯 지속 지원**: True
   - 아임웹이 코드위젯 기능을 계속해서 지원할 것으로 가정

2. **Cloudflare Pages 안정성**: 가정 (항상 true는 아님)
   - Cloudflare Pages가 안정적으로 작동할 것으로 가정하나, 일시적 장애 가능성 존재

3. **Shadow DOM 브라우저 호환성**: True (검증됨)
   - 모든 주요 브라우저(Chrome, Firefox, Safari, Edge)에서 Shadow DOM이 정상 작동함

4. **개인정보 보호 및 보안**:
   - 입력폼이 수집하는 이름, 전화번호 등 개인정보에 대한 보안 대책 필요
   - Airtable 데이터베이스 보안 및 접근 제어 필요
   - SSL/TLS를 통한 암호화된 데이터 전송

5. **클라이언트 수용성**:
   - 클라이언트가 웹 컴포넌트 기반의 새로운 디자인 스타일을 수용할 것으로 가정

## 리스크 및 대응 방안 (Risks and Mitigation)

### 우선순위 1: 실제 제작 시간이 목표를 초과

- **리스크**: 웹 컴포넌트 시스템 도입 후에도 실제 랜딩페이지 제작 시간이 목표(원페이지 4시간, 멀티페이지 10시간)를 초과할 가능성
- **가능성**: 높음
- **영향도**: 매우 높음 (프로젝트의 핵심 목표)
- **대응 방안**:
  - **예방**: 이 프로젝트 자체가 이 문제를 해결하기 위해 설계됨
  - **모니터링**: 초기 몇 개 프로젝트에서 제작 시간을 철저히 추적
  - **개선**: 병목 지점을 파악하여 워크플로우 지속적으로 개선
  - **조정**: 필요시 컴포넌트 기능 단순화 또는 config 구조 최적화

### 우선순위 2: 구글애즈 전환추적 시 이벤트 추적 실패

- **리스크**: GTM/GA4/구글애즈 이벤트가 정상적으로 송출되지 않거나 추적되지 않을 가능성
- **가능성**: 높음 (새로운 시스템 통합)
- **영향도**: 높음 (마케팅 성과 측정 불가)
- **대응 방안**:
  - **예방**: 전환추적 방법을 사전에 철저히 개발 및 테스트
  - **검증**: GTM 미리보기 모드를 통한 실시간 이벤트 확인
  - **테스트**: 실제 제출 시나리오를 반복 테스트
  - **백업**: dataLayer 직접 푸시 방식과 gtag 방식 모두 준비
  - **문서화**: 이벤트 명세서 작성 및 디버깅 가이드 준비

### 우선순위 3: 오늘 안에 MVP 완성 못하는 경우

- **리스크**: 예상보다 개발이 지연되어 오늘 안에 MVP(멀티페이지 1개)를 완성하지 못할 가능성
- **가능성**: 중간
- **영향도**: 중간 (일정 지연)
- **대응 방안**:
  - **예방**: 기능을 우선순위별로 나누어 핵심 기능 먼저 구현
  - **집중**: 오늘은 다른 업무를 최소화하고 개발에 집중
  - **유연성**: 필요시 야근을 통해 데드라인 준수
  - **범위 조정**: 최악의 경우 히어로 애니메이션 등 부가 기능을 Day 2로 연기

### 우선순위 4: 개인정보 유출 및 Airtable 보안 문제

- **리스크**: 입력폼으로 수집한 개인정보(이름, 전화번호)가 유출되거나 Airtable이 해킹당할 가능성
- **가능성**: 낮음-중간
- **영향도**: 매우 높음 (법적 문제, 신뢰 손실)
- **대응 방안**:
  - **보안 대책 구상 필요**:
    1. HTTPS 강제 적용 (모든 데이터 전송 암호화)
    2. Airtable API 키 환경변수 처리 (코드에 직접 노출 금지)
    3. Airtable 접근 제어 설정 (IP 화이트리스트, 권한 최소화)
    4. 개인정보 처리방침 명시 및 동의 절차 강화
    5. 정기적인 보안 감사 및 로그 모니터링
    6. 데이터 보관 기간 설정 및 자동 삭제 정책

### 우선순위 5: 오너가 config 사용법을 이해 못하는 경우

- **리스크**: 비개발자인 오너가 config 파일의 구조와 사용법을 이해하지 못해 수정 불가능할 가능성
- **가능성**: 낮음-중간
- **영향도**: 중간 (개발자 의존성 지속)
- **대응 방안**:
  - **예방**: config 구조를 최대한 직관적이고 단순하게 설계
  - **문서화**: 주석을 상세히 작성하여 각 항목의 의미 명확히 설명
  - **교육**: 오너에게 config 수정 방법을 단계별로 교육
  - **예시**: 실제 사용 예시를 여러 개 준비하여 참고 가능하도록 함
  - **지원**: 초기에는 같이 작업하며 익숙해질 수 있도록 지원
