## 문서 개요

이 PRD (Product Requirements Document)는 100zoad 웹 컴포넌트 시스템의 기능 요구사항과 비기능 요구사항을 정의합니다. 무엇을 만들지에 대한 단일 진실 공급원(Single Source of Truth)이며, 요구사항부터 구현까지의 추적성을 제공합니다.

**관련 문서:**
- 제품 브리프: docs/product-brief-100zoad-2026-01-01.md

---

## 요약 (Executive Summary)

부동산 분양 홍보 랜딩페이지를 위한 규격화된 웹 컴포넌트 시스템을 개발합니다. Cloudflare Pages와 Shadow DOM을 활용하여 아임웹 기반 웹디자인 및 개발 시간을 대폭 단축하고, 코드 지식이 없는 팀원도 안전하게 스타일과 문구를 수정할 수 있도록 합니다. 온라인 마케팅 대행사의 2명 팀(오너, 개발자/디자이너)이 사용하며, 클라이언트 1인당 맨아워를 줄이고 저비용 소규모 비즈니스 모델을 유지하는 것이 핵심 목표입니다.

## 제품 목표 (Product Goals)

### 비즈니스 목표

1. **제작 시간 대폭 단축**
   - 멀티페이지: 일주일 → 10시간 (93% 단축)
   - 원페이지: 4-5시간 → 4시간 (20% 단축)
   - 복잡한 원페이지: 8시간 → 4시간 (50% 단축)

2. **생산성 향상**
   - 멀티페이지: 2달에 1개 → 월 5개 (10배 증가)
   - 원페이지: 월 7개 → 월 15개 (2배 증가)

3. **업무 배분 최적화**
   - 랜딩페이지 제작: 월 15일
   - 다른 업무 (마케팅 관리, 애셋 제작 등): 월 15일

4. **저비용 소규모 비즈니스 모델 유지**
   - 추가 인력 채용 없이 생산성 향상
   - 2명 팀으로 경쟁력 유지

### 성공 지표

**주요 지표:**
- 랜딩페이지 제작 시간 추적 (멀티페이지, 원페이지 각각)
- 월별 랜딩페이지 생산량 (멀티페이지 5개, 원페이지 15개 목표)

**보조 지표 (장기적):**
- 리드 전환율 (데이터가 충분히 쌓인 후 의미 있음)

**정량적 목표:**
- 원페이지 제작 시간: 4시간 이내
- 멀티페이지 제작 시간: 10시간 이내
- MVP 출시: 2026-01-01 (오늘)
- 안정화 완료: 2026-01-05 (4일 후)

## 기능 요구사항 (Functional Requirements)

기능 요구사항(FR)은 시스템이 **무엇을** 해야 하는지 정의합니다.

각 요구사항 포함 내용:
- **ID**: 고유 식별자 (FR-001, FR-002 등)
- **Priority**: Must Have / Should Have / Could Have (MoSCoW 방법)
- **Description**: 시스템이 수행해야 할 작업
- **Acceptance Criteria**: 완료 검증 방법

---

### 헤더 컴포넌트

#### FR-001: PC 헤더 메뉴 표시

**Priority:** Must Have

**Description:**
PC 화면에서 헤더에 최대 6개의 메뉴 항목을 표시합니다. 메뉴 구성, 색상, 로고는 main.html config로 조정 가능합니다.

**Acceptance Criteria:**
- [ ] 최대 6개 메뉴 항목 지원
- [ ] 로고 이미지 표시 (512x512px PNG, 아임웹 게시판 업로드 후 URL 삽입)
- [ ] config로 메뉴 텍스트, URL, 색상 조정 가능
- [ ] 스크롤 시 헤더 상단 고정 (fixed position)

---

#### FR-002: 관심고객등록 메뉴 (특수 메뉴)

**Priority:** Must Have

**Description:**
헤더 맨 오른쪽에 "관심고객등록" 메뉴를 고정하고, CSS 애니메이션으로 시선을 끕니다. 클릭 시 입력폼으로 이동합니다.

**Acceptance Criteria:**
- [ ] 맨 오른쪽 메뉴로 고정
- [ ] CSS 애니메이션 적용 (config로 애니메이션 타입 조정 가능)
- [ ] 원페이지: 입력폼 섹션으로 부드러운 스크롤 (아임웹 섹션 ID/CSS 선택자 기반)
- [ ] 멀티페이지: 입력폼 페이지로 페이징
- [ ] config로 타겟 섹션 ID/URL 지정 가능

---

#### FR-003: 헤더 페이지 타입별 동작

**Priority:** Must Have

**Description:**
원페이지와 멀티페이지에서 헤더 메뉴 클릭 시 동작이 다릅니다.

**Acceptance Criteria:**
- [ ] 원페이지: 앵커 링크로 섹션 스크롤
- [ ] 멀티페이지: 페이지 이동 (URL 기반)
- [ ] config의 pageType 설정으로 동작 전환
- [ ] 부드러운 스크롤 애니메이션 (duration config 가능)

---

#### FR-004: 모바일 햄버거 메뉴

**Priority:** Must Have

**Description:**
모바일 화면에서 헤더는 햄버거 메뉴로 표시되며, 펼쳤을 때 상단에 로고가 표시됩니다.

**Acceptance Criteria:**
- [ ] 모바일 브레이크포인트(기본: 768px) 이하에서 햄버거 아이콘 표시
- [ ] 햄버거 클릭 시 사이드/전체 메뉴 펼침
- [ ] 펼친 메뉴 상단에 로고 표시
- [ ] 메뉴 항목 클릭 시 메뉴 자동 닫힘
- [ ] config로 햄버거 위치, 메뉴 스타일 조정 가능

---

#### FR-005: 파비콘 설정

**Priority:** Should Have

**Description:**
헤더 로고 이미지를 파비콘으로도 사용합니다.

**Acceptance Criteria:**
- [ ] config의 로고 URL을 파비콘으로 자동 설정

- [ ] 브라우저 탭에 파비콘 표시

## 비기능 요구사항 (Non-Functional Requirements)

비기능 요구사항(NFR)은 시스템이 **어떻게** 작동해야 하는지 정의합니다 - 품질 속성 및 제약사항.

---

### Performance (성능)

#### NFR-001: 초기 로딩 시간

**Priority:** Must Have

**Description:**
페이지 초기 로딩 시간은 3초 이내여야 하며, 로딩 중 디자인된 로딩 화면을 표시하여 UX 손실을 방지합니다.

**Acceptance Criteria:**
- [ ] 모든 컴포넌트 초기 로딩 완료: 3초 이내
- [ ] 로딩 화면 표시 (스피너, 로고, 진행바 등 - config로 설정)
- [ ] 컴포넌트 개별 로딩 시간: 각 500ms 이내
- [ ] Lighthouse Performance 점수: 90 이상

**Rationale:**
랜딩페이지는 첫인상이 중요하며, 로딩 지연은 이탈률 증가로 이어집니다.

---

#### NFR-002: 런타임 성능

**Priority:** Must Have

**Description:**
사용자 여정 중 컴포넌트 동작(스크롤, 클릭, 입력폼 제출)이 즉각적으로 반응해야 합니다.

**Acceptance Criteria:**
- [ ] 버튼 클릭 반응 시간: 100ms 이내
- [ ] 입력폼 제출 응답: 2초 이내 (Airtable 전송 포함)
- [ ] 스크롤 애니메이션 부드러움: 60fps 유지
- [ ] 메모리 누수 없음 (장시간 페이지 유지 시)

**Rationale:**
사용자 여정 중 로딩으로 인한 UX 손상 방지가 핵심 요구사항입니다.

---

### Security (보안)

#### NFR-003: 데이터 전송 암호화

**Priority:** Must Have

**Description:**
모든 개인정보 전송은 HTTPS를 통해 암호화됩니다.

**Acceptance Criteria:**
- [ ] HTTPS 강제 적용 (HTTP 접속 시 HTTPS로 리다이렉트)
- [ ] SSL/TLS 인증서 적용 (Cloudflare 제공)
- [ ] Airtable API 호출 HTTPS 사용
- [ ] Mixed Content 오류 없음

**Rationale:**
개인정보보호법 준수 및 사용자 신뢰 확보

---

#### NFR-004: API 키 보안

**Priority:** Must Have

**Description:**
Airtable API 키 및 민감 정보는 코드에 직접 노출되지 않습니다.

**Acceptance Criteria:**
- [ ] API 키는 환경변수 또는 서버리스 함수로 관리
- [ ] 클라이언트 코드에서 API 키 직접 노출 금지
- [ ] config 예시 문서에 실제 키 포함 금지 (플레이스홀더 사용)
- [ ] .env 파일은 .gitignore에 포함

**Rationale:**
API 키 유출 시 Airtable 무단 접근 및 데이터 유출 위험

---

#### NFR-005: 개인정보 처리 규정 준수

**Priority:** Must Have

**Description:**
개인정보보호법에 따른 개인정보 수집, 저장, 처리 규정을 준수합니다.

**Acceptance Criteria:**
- [ ] 이용약관 및 개인정보처리방침 명시적 동의
- [ ] 필수 동의 없이는 제출 불가
- [ ] 수집 목적, 항목, 보유기간 명시
- [ ] Airtable 데이터 접근 제어 설정 (IP 화이트리스트 권장)
- [ ] 향상된 전환 추적 시 개인정보 해시화

**Rationale:**
법적 준수 및 사용자 신뢰 확보

---

### Reliability/Availability (신뢰성/가용성)

#### NFR-006: 컴포넌트 로딩 실패 시 폴백 (Graceful Degradation)

**Priority:** Must Have

**Description:**
Cloudflare Pages 장애 시 하이브리드 전략으로 핵심 기능을 보장합니다.

**Acceptance Criteria:**
- [ ] 입력폼 로딩 실패 시: 간단한 HTML form 폴백 (아임웹 내장)
- [ ] 헤더 로딩 실패 시: 로고 + 전화번호만 표시
- [ ] 퀵메뉴/모바일 내비게이터 로딩 실패 시: 숨김 처리
- [ ] 컴포넌트별 타임아웃: 3초
- [ ] 로딩 실패 시 콘솔 로그 기록
- [ ] 핵심 기능(리드 수집, 전화걸기) 항상 동작

**Rationale:**
Cloudflare Pages 일시 장애 시에도 비즈니스 연속성 보장

---

#### NFR-007: 입력폼 제출 오류 처리

**Priority:** Must Have

**Description:**
입력폼 제출 실패 시 사용자에게 명확한 오류 메시지를 표시합니다.

**Acceptance Criteria:**
- [ ] 네트워크 오류, Airtable API 오류 구분하여 메시지 표시
- [ ] 오류 메시지 config로 커스터마이징 가능
- [ ] 재제출 버튼 제공
- [ ] 입력 데이터 유지 (사용자가 다시 입력할 필요 없음)
- [ ] 오류 로그 콘솔에 기록

**Rationale:**
제출 실패는 로직 문제일 가능성이 높으므로, 명확한 오류 정보 제공이 중요

---

### Usability (사용성)

#### NFR-008: 비개발자 Config 수정 가능성

**Priority:** Must Have

**Description:**
코드 지식이 없는 오너가 config를 수정하여 디자인과 문구를 변경할 수 있어야 합니다.

**Acceptance Criteria:**
- [ ] Config 구조가 직관적 (명확한 키 이름, 계층 구조)
- [ ] 모든 config 항목에 한글 주석 설명
- [ ] 실제 사용 예시 여러 개 제공
- [ ] config 오류 시 명확한 오류 메시지 (어떤 항목이 잘못되었는지)
- [ ] 오너 교육 가이드 문서 제공

**Rationale:**
개발자 부재 시에도 수정 가능해야 비즈니스 연속성 확보

---

### Maintainability (유지보수성)

#### NFR-009: 코드 품질 및 문서화

**Priority:** Should Have

**Description:**
코드는 읽기 쉽고 유지보수 가능하며, 충분히 문서화되어야 합니다.

**Acceptance Criteria:**
- [ ] 모든 함수에 JSDoc 주석
- [ ] 컴포넌트별 README 파일
- [ ] Config 스키마 문서
- [ ] 배포 가이드 문서
- [ ] 이벤트 추적 명세서

**Rationale:**
혼자 개발하지만, 미래의 자신 또는 인수인계를 위한 문서화 필요

---

### Compatibility (호환성)

#### NFR-010: 아임웹 코드위젯 호환성

**Priority:** Must Have

**Description:**
모든 컴포넌트가 아임웹 코드위젯 환경에서 정상 작동해야 합니다.

**Acceptance Criteria:**
- [ ] 아임웹 코드위젯 내 JavaScript 실행 정상
- [ ] 아임웹 기본 스타일과 충돌 없음 (Shadow DOM 격리)
- [ ] 아임웹 섹션 ID/CSS 선택자 기반 스크롤 정상 작동
- [ ] 아임웹 게시판 업로드 이미지 URL 정상 로드
- [ ] 아임웹 페이지 네비게이션 정상 작동

**Rationale:**
아임웹 플랫폼 의존성이 핵심 제약사항

## Epics

Epics는 관련 기능의 논리적 그룹핑이며, Sprint Planning (Phase 4)에서 사용자 스토리로 세분화됩니다.

각 Epic은 여러 기능 요구사항과 매핑되며, 2-10개의 스토리를 생성합니다.

---

### EPIC-001: 헤더 컴포넌트

**Description:**
PC와 모바일 환경에서 작동하는 헤더 컴포넌트를 개발합니다. 원페이지/멀티페이지 모두 지원하며, 특수 메뉴("관심고객등록")를 포함합니다.

**Functional Requirements:**
- FR-001: PC 헤더 메뉴 표시
- FR-002: 관심고객등록 메뉴
- FR-003: 헤더 페이지 타입별 동작
- FR-004: 모바일 햄버거 메뉴
- FR-005: 파비콘 설정

**Story Count Estimate:** 5-7

**Priority:** Must Have

**Business Value:**
헤더는 모든 페이지에서 사용자가 가장 먼저 보는 요소로, 브랜드 인식과 내비게이션의 핵심입니다. "관심고객등록" 특수 메뉴는 리드 전환의 주요 진입점입니다.

---

### EPIC-002: 푸터 컴포넌트

**Description:**
2단 레이아웃의 푸터 컴포넌트를 개발합니다. 현장정보와 대행사정보를 표시하며, 법적 정보 링크를 포함합니다.

**Functional Requirements:**
- FR-006: 2단 푸터 레이아웃
- FR-007: 법적 정보 링크
- FR-008: 푸터 반응형 디자인

**Story Count Estimate:** 3-4

**Priority:** Must Have

**Business Value:**
푸터는 신뢰성과 법적 준수를 보여주는 중요한 요소입니다. 현장정보와 대행사정보를 명확히 구분하여 표시함으로써 전문성을 전달합니다.

---

### EPIC-003: 입력폼 컴포넌트 및 리드 관리

**Description:**
동적 필드 구성이 가능한 입력폼 컴포넌트를 개발하고, Airtable 연동을 통한 리드 수집 및 전송 기능을 구현합니다. 팝업과 섹션 두 가지 표시 방식을 지원합니다.

**Functional Requirements:**
- FR-009: 필수 필드 (이름, 전화번호)
- FR-010: 선택 필드 동적 구성
- FR-011: 전화번호 실시간 하이픈 삽입
- FR-012: 법적 동의 체크박스
- FR-013: 입력폼 제출 및 성공 메시지
- FR-014: 입력폼 두 가지 표시 방식
- FR-024: Airtable 리드 전송

**Related NFRs:**
- NFR-003: 데이터 전송 암호화
- NFR-005: 개인정보 처리 규정 준수
- NFR-007: 입력폼 제출 오류 처리

**Story Count Estimate:** 8-10

**Priority:** Must Have

**Business Value:**
입력폼은 비즈니스 목표(리드 수집)의 핵심 기능입니다. 동적 필드 구성으로 다양한 클라이언트 요구사항에 유연하게 대응할 수 있습니다.

---

### EPIC-004: 퀵메뉴 및 모바일 내비게이터

**Description:**
PC용 퀵메뉴와 모바일용 하단 내비게이터를 개발합니다. 각 버튼은 입력폼 팝업, 전화걸기, URL 이동 등의 액션을 수행합니다.

**Functional Requirements:**
- FR-015: PC 퀵메뉴 표시
- FR-016: 퀵메뉴 버튼 액션

- FR-018: 모바일 내비게이터 버튼 액션

**Story Count Estimate:** 4-6

**Priority:** Must Have

**Business Value:**
퀵메뉴와 내비게이터는 사용자가 페이지 어디에서든 즉시 행동(리드 등록, 전화)할 수 있도록 하는 전환 가속화 요소입니다.

**Functional Requirements:**
- FR-019: 중앙 집중식 Config 구조

**Priority:** Must Have

**Business Value:**

### EPIC-006: 이벤트 추적 및 마케팅 분석

**Description:**
GTM/GA4/구글애즈 이벤트 통합을 구현합니다. 전환 이벤트(입력폼 제출, 전화걸기)와 비전환 이벤트(메뉴 클릭, 스크롤 깊이)를 추적합니다.

**Functional Requirements:**
- FR-022: 전환 이벤트 추적
- FR-023: 비전환 이벤트 추적

**Story Count Estimate:** 3-5

**Priority:** Must Have

**Business Value:**
마케팅 성과 측정은 비즈니스 ROI 판단의 핵심입니다. 향상된 전환 추적을 통해 구글애즈 최적화가 가능하며, 이는 클라이언트 만족도에 직결됩니다.

---

### EPIC-007: 반응형 디자인 및 크로스 브라우저 호환성

**Description:**
PC와 모바일 환경에서 최적화된 레이아웃을 제공하고, 모든 주요 브라우저에서 정상 작동하도록 합니다.

**Functional Requirements:**
- FR-025: PC/모바일 반응형 전환
- FR-026: 크로스 브라우저 호환성

**Story Count Estimate:** 2-4

**Priority:** Must Have

**Business Value:**
모바일 트래픽 비중이 높은 부동산 분양 랜딩페이지에서 반응형 디자인은 필수입니다. 브라우저 호환성은 사용자 이탈을 방지합니다.

---

### EPIC-008: 성능, 보안 및 안정성 구현

**Description:**

## 주요 사용자 플로우 (User Flows)

### Flow 1: 리드 등록 (헤더 경로)

1. 사용자가 랜딩페이지 방문
2. 헤더의 "관심고객등록" 메뉴 클릭 (CSS 애니메이션으로 시선 유도)
3. **원페이지:** 입력폼 섹션으로 부드러운 스크롤 / **멀티페이지:** 입력폼 페이지로 이동
4. 필수 필드 입력 (이름, 전화번호) + 선택 필드 입력
5. 전화번호 입력 시 실시간 하이픈 자동 삽입
6. 이용약관/개인정보처리방침 체크박스 선택 (필수)
7. 제출 버튼 클릭
8. Airtable로 데이터 전송 (HTTPS 암호화)
9. 성공 메시지 팝업 표시
10. GTM/GA4 전환 이벤트 송출 (향상된 전환 추적 포함)

**예상 소요 시간:** 1-2분

---

### Flow 2: 리드 등록 (퀵메뉴/내비게이터 경로)

1. 사용자가 페이지 스크롤 중
2. **PC:** 우측 퀵메뉴의 "입력폼 팝업" 버튼 클릭 / **모바일:** 하단 내비게이터의 "입력폼 팝업" 버튼 클릭
3. 입력폼 모달 팝업 표시
4. 필드 입력 및 제출 (Flow 1의 4-10단계 동일)
5. 팝업 확인 후 폼 초기화 및 팝업 닫김

**예상 소요 시간:** 1-2분

**차별점:** 페이지 이동 없이 즉시 등록 가능 (전환율 향상)

---

### Flow 3: 전화 문의

1. 사용자가 전화 상담 희망
2. **PC:** 우측 퀵메뉴의 "전화걸기" 버튼 클릭 / **모바일:** 하단 내비게이터의 "전화걸기" 버튼 클릭
3. **모바일:** tel: 링크로 전화 앱 자동 실행 / **PC:** 전화번호 표시 또는 클립보드 복사
4. GTM/GA4 전환 이벤트 송출 (call_click)

**예상 소요 시간:** 즉시

**차별점:** 원클릭 전화 연결로 전환 장벽 최소화

### 외부 의존성

1. **Cloudflare Pages**
   - 역할: 웹 컴포넌트 호스팅 및 CDN 배포
   - 플랜: 무료 플랜
   - 제약: 빌드 500회/월 (충분), 요청/대역폭 무제한
   - 리스크: 일시 장애 가능성 (NFR-006 폴백 전략으로 대응)

2. **Airtable**
   - 역할: 리드 데이터 저장 및 관리
   - 플랜: Team 플랜
   - 제약: API Rate Limit 50 req/s, Automation 100,000회/월
   - 용량: 월 1,000 리드 처리 충분 (2% 사용률)

3. **아임웹**
   - 역할: 랜딩페이지 플랫폼 (코드위젯, 섹션 구조, 게시판)
   - 플랜: 프로 플랜
   - 제약: 트래픽 제한 확인 필요 (월 100,000 방문자)
   - 리스크: 트래픽 초과 시 추가 비용 또는 차단

4. **Google Tag Manager (GTM)**
   - 역할: 이벤트 태그 관리
   - 플랜: 무료
   - 제약: 없음

5. **Google Analytics 4 (GA4)**
   - 역할: 웹 분석 및 전환 데이터 수집
   - 플랜: 무료
   - 제약: 월 1000만 이벤트 (충분)

6. **Google Ads**
   - 역할: 전환 추적 및 마케팅 캠페인
   - 플랜: 광고 집행에 따른 비용
   - 제약: 없음

7. **Make.com**
   - 역할: 리드 전송 워크플로우 자동화 (Airtable → 다른 시스템)
   - 플랜: 사용자 구독 플랜
   - 제약: 플랜별 작업 수 제한

## 가정사항 (Assumptions)

1. **아임웹 코드위젯 지속 지원: True**
   - 아임웹이 코드위젯 기능을 계속해서 지원할 것으로 가정

2. **Cloudflare Pages 안정성: 가정 (항상 true는 아님)**
   - Cloudflare Pages가 안정적으로 작동할 것으로 가정하나, 일시적 장애 가능성 존재
   - 대응: NFR-006 폴백 전략 (Graceful Degradation)

3. **Shadow DOM 브라우저 호환성: True (검증됨)**
   - 모든 주요 브라우저(Chrome, Firefox, Safari, Edge)에서 Shadow DOM이 정상 작동함

4. **개인정보 보호 및 보안:**
   - 입력폼이 수집하는 이름, 전화번호 등 개인정보에 대한 보안 대책 필요
   - Airtable Team 플랜으로 데이터베이스 보안 및 접근 제어 가능
   - SSL/TLS를 통한 암호화된 데이터 전송

5. **클라이언트 수용성:**
   - 클라이언트가 웹 컴포넌트 기반의 새로운 디자인 스타일을 수용할 것으로 가정

6. **용량 충분성:**
   - Airtable Team 플랜 (100,000 automation/월, 50 req/s)으로 월 1,000 리드 처리 충분
   - Cloudflare Pages 무료 플랜으로 월 100,000 방문자 처리 충분
   - 아임웹 프로 플랜으로 트래픽 처리 가능 (확인 필요)

### 1. 아임웹 프로 플랜 트래픽 제한 (우선순위: 높음)

**질문:** 아임웹 프로 플랜이 월 100,000 방문자를 감당할 수 있는가?

**필요 정보:**
- 프로 플랜 월 PV(페이지뷰) 제한
- 프로 플랜 월 UV(순 방문자) 제한
- 하나의 계정에서 관리 가능한 사이트 수 (월 20개 랜딩페이지)
- 트래픽 초과 시 요금 정책

**해결 방법:** 아임웹 고객센터 문의

**영향:** 트래픽 제한 초과 시 추가 비용 또는 서비스 중단 위험

---

### 2. Cloudflare Pages 프로젝트 구조 (우선순위: 중간)

**질문:** 모든 클라이언트 컴포넌트를 하나의 Cloudflare Pages 프로젝트에서 호스팅할 것인가, 아니면 클라이언트별로 별도 프로젝트를 생성할 것인가?

**옵션 A: 단일 프로젝트 (권장)**
- 장점: 배포 1회로 모든 클라이언트 업데이트, 관리 간편
- 단점: 프로젝트 구조 복잡도 증가
- 구조 예시:
  ```
  your-components-project/
  ├── common/
  │   ├── header.js
  │   ├── footer.js
  │   └── form.js
  └── assets/logos/
  ```

**옵션 B: 클라이언트별 프로젝트**
- 장점: 클라이언트별 독립성
- 단점: 배포 관리 복잡, 여러 무료 플랜 필요

**해결 방법:** 아키텍처 설계 단계(Phase 3)에서 결정

---

### 3. Airtable API 호출 방식 (우선순위: 중간)

**질문:** 클라이언트 사이드에서 직접 Airtable API 호출할 것인가, 아니면 서버리스 함수(Cloudflare Workers 등)를 경유할 것인가?

**옵션 A: 클라이언트 직접 호출**
- 장점: 구현 간단
- 단점: API 키 노출 위험 (config에서 읽어야 함)

**옵션 B: 서버리스 함수 경유 (권장)**
- 장점: API 키 안전, 추가 로직 가능 (스팸 방지 등)
- 단점: Cloudflare Workers 설정 필요

**해결 방법:** 아키텍처 설계 단계(Phase 3)에서 결정
